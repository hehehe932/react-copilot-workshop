why-react.mdphase 1
Why React?
React flips the UI update model from manual changes to declarative state.
Before React
Manually update the page
button.onclick = () => {
  count++;
  label.innerText = count;
}

You find elements and tell the browser exactly what to change.

With React
Update state, React updates UI
setCount(count + 1)

<p>{count}</p>

You describe the UI for the current data. React handles the screen update.

Remember

Declarative UI means you write what you want to see based on state. You update the state, and React handles the DOM.

React Component Mental Model
A component is a reusable UI block. For this app, the tree is simple.
App<- holds tasks state, calculates counts
Header<- displays logo and "X of Y tasks done"

Remember

Think of the tree as UI ownership: App manages shared data, while smaller child components focus on one visual job. Data flows down to children; click events travel back up through callback props.

JSX
JSX looks like HTML, but it is written inside JavaScript.
What this means

JSX lets structure and JavaScript values live together. Use curly braces when the UI needs data such as a name, count, or task title.

Rules of JSX
{ }Use curly braces for JavaScript variables or logic
HTMLclass="card" becomes className="card"
Tags<input> becomes <input />
1Must return a single parent element (or a fragment <>)

function App() {
  const name = "Alya";

  return (
    <main>
      <h1 className="header-title">Student Task Manager</h1>
      <p>Built by {name}</p>
    </main>
  );
}

state.jsxphase 1
State: React's Memory
State stores data that can change and trigger a re-render.
What this means

State is React’s memory. When it changes through its setter, React re-renders the UI so the screen matches the latest data.

taskscurrent value
setTasksupdate function
[]starting value
Remember

Never mutate state directly. Create a new array and pass it to the setter so React can detect the update.

tasks.push(newTask) // wrongsetTasks([...tasks, newTask]) // right
App.jsx
import { useState } from 'react';

const [tasks, setTasks] = useState([]);

// tasks = current value
// setTasks = update function
// [] = starting value

Controlled input plus a state update.

The input is controlled by state: typing updates newTask, and submitting creates a new task object for the tasks array.

Remember

The handler has three jobs: reject blank input after trim(), add a task in a fresh array, then clear the input.

Complete AddTaskForm.jsx

Open src/components/AddTaskForm.jsx. Accept value, onChange, and onAdd. Update the input to use onChange={(e) => onChange(e.target.value)} and the button to trigger onAdd.

valuecomes from newTask
onChangecalls setNewTask
onAddruns handleAddTask
App.jsx
const [newTask, setNewTask] = useState("");

const handleAddTask = () => {
  // 1. Reject empty input
  if (!newTask.trim()) return;

  // 2. Add new task to the array
  setTasks([
    ...tasks,
    {
      id: Date.now(),
      title: newTask.trim(),
      status: 'Pending',
      priority: 'medium'
    }
  ]);

  // 3. Clear the input
  setNewTask("");
};

App.jsxphase 1
Delete Task: The onDelete Prop
Remove an item without mutating the original array.
Remember

filter() checks each id and returns a new array without the selected task. The original array is not mutated.

The onDelete Prop

The button lives in TaskCard, but the state is in App. We pass the delete handler via the onDelete prop to TaskCard, so clicking the button calls the parent function with the task's ID.

[A, B, C]filter out B ->[A, C]
delete-task.jsx
const handleDeleteTask = (id) => {
  setTasks(
    tasks.filter(task => task.id !== id)
  );
};

<TaskCard
  onDelete={handleDeleteTask}
/>

complete-counter.jsxphase 1
Complete Task: The onToggle Prop
Two connected ideas: update one task, then calculate live counts from the updated array.
Toggle Complete
handleToggleTask
const handleToggleTask = (id) => {
  setTasks(tasks.map(task =>
    task.id === id
      ? { ...task, status: task.status === 'Done'
        ? 'Pending' : 'Done' }
      : task
  ));
};

<TaskCard onToggle={handleToggleTask} />
The onToggle Prop

We pass the handler via the onToggle prop. map() flips status (Pending ↔ Done) for the matching task and keeps others unchanged.

prompts.mdphase 2
Writing Better Prompts
Give Copilot specific, high-quality context for better suggestions.
Weak Prompt
// add filter
Too brief. Copilot does not know the inputs, behavior, or constraints.
Strong Prompt
// Add a search bar that filters tasks by title and status in real time. Keep the original tasks array unchanged.
Clear task, inputs, expected behavior, and constraint.
What to build (e.g. Search input) Data you have (e.g. tasks, searchTerm) Expected behavior (e.g. filter array) Constraints (e.g. case insensitive)
Mental model

A good prompt is a tiny specification: name the task, the available data, the expected result, and anything the solution must avoid.
